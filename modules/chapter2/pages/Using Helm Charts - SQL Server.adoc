= Using Helm Charts - SQL Server

Operators and Helm are not mutually exclusive; in a mature OpenShift environment, they often function as a layered stack. While Helm is the best tool for "Day 1" (packaging and templating), Operators are the masters of "Day 2" (ongoing state management).

In many organizations, the choice depends on the nature of the workload:

* *Use Helm for Stateless Apps*: For simple web frontends or APIs where "Day 2" is just "restart if it crashes," Helm is more than enough. It provides versioning, easy rollbacks, and environment-specific variable injection.

* *Use Operators for Stateful Apps*: For databases (Postgres, MongoDB) or complex middleware (Kafka/Confluent), an Operator is essential. It understands the application's internal stateâ€”knowing, for instance, that it shouldn't reboot a database node while a backup is in progress.

This is just an example of a Helm chart for SQL Server for illustration purposes. It is based on Microsoft official documentation to deploy SQL Server container clusters.

First, create a new project to deploy SQL Server. Go to `Home` -> `Projects` and create a new project.

Go to `Helm` -> `Repositories` and click on `Create Helm Repository`    .

Fill out the following data:

* Scope Type: `Cluster scoped`
* Name: `sqlserver-helm-charts`
* URL: https://mpbravo.github.io/helm-charts/

image::create-helm-repository.png[Create Helm Repository,1000,1000]

Go to `Ecosystem` -> `Software Catalog`

Filter by `Helm Charts`, and in `Helm Repositories`, select the one we have just created. The SQL Server should show up.

image::helm-charts.png[Software Catalog Helm Charts,1000,1000]

Click on the SQL Server Helm Chart and click on `Create`.

Expand the form and add the SA password for the new database. Then click on `Create`.

image::create-helm-release.png[Create SQL Server,1000,1000]

If everything is ok, then you will be redirected to the Topology view with the SQL Server pod running.

You can connect to the SQL Server instance through node ports, using one of the nodes IP/hostname and the port showing here:

--
[source,bash]
----
> oc get services
NAME               TYPE           CLUSTER-IP      EXTERNAL-IP   PORT(S)          AGE
mssql-deployment   LoadBalancer   172.30.38.133   <pending>     1433:30701/TCP   14m
> sqlcmd -S node-hostname,30701 -U sa -P redhat123. -C
----
--

NOTE: node ports are disallowed in most customers, this is just an example to check connection to the database.
